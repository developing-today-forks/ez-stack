# ez-stack v0.1.5 — Agent UX + Human Convenience Features

**Date:** 2026-03-11
**Status:** Approved

## Background

Two input sources drove this release:
1. Feature ideas from daily human use (`ez-feature-ideas.md`)
2. Direct friction report from an AI agent (Claude) using ez in a long coding session

The highest-leverage improvements are agent-facing: `--autostash`, `--json` output, non-interactive
`checkout`, `--from` on create, and structured exit codes. Human-facing additions include `ez draft`/`ez ready`, `ez pr-link`, and `ez pr-edit` opening `$EDITOR`.

---

## Features

### 1. `ez sync --autostash`

`ez sync` currently fails with a git error if there are unstaged or uncommitted changes (because `git checkout <trunk>` refuses to switch branches). This is the most common agent failure mode.

**Behavior:**
- New `--autostash` flag on `Sync`
- Before any mutation: `git stash push --include-untracked -m "ez-autostash"` — only if there are dirty changes
- After sync completes (success or failure): `git stash pop` if we stashed
- The stash pop happens in a cleanup path so it runs even when sync returns early via `?`

**Implementation notes:**
- `git.rs`: `stash_push() -> Result<bool>` (true = something was stashed), `stash_pop() -> Result<()>`
- `sync.rs`: accepts `autostash: bool`; RAII-style cleanup via explicit `stash_pop()` before every return path
- `cli.rs`: add `#[arg(long)] autostash: bool` to `Sync`

### 2. `--json` output on `ez status` and `ez log`

Human-readable colored output is hard to parse for agents. `--json` prints machine-readable JSON to **stdout** (not stderr) with no color codes.

**`ez status --json` output schema:**
```json
{
  "branch": "feat/x",
  "parent": "main",
  "pr_number": 42,
  "pr_url": "https://github.com/owner/repo/pull/42",
  "pr_state": "OPEN",
  "is_draft": false,
  "depth": 2,
  "commits": 1,
  "children": ["feat/y"],
  "needs_restack": false
}
```

**`ez log --json` output schema:** Flat array (not nested), one entry per branch (trunk excluded):
```json
[
  {
    "branch": "feat/a",
    "parent": "main",
    "depth": 1,
    "pr_number": 40,
    "pr_url": "...",
    "pr_state": "OPEN",
    "is_draft": false,
    "commits": 2,
    "children": ["feat/b"],
    "needs_restack": false
  }
]
```

**Implementation notes:**
- No new dependencies — use `serde_json::json!`
- Both commands check the flag; if set, collect data, `println!` JSON, return early

### 3. `ez checkout <name>` — non-interactive direct switch

`ez checkout` currently always opens a TUI selector, blocking agents that don't have a TTY.

**Behavior:**
- `Checkout` variant gains `name: Option<String>` positional arg
- If provided as a branch name: validate it's managed or trunk, call `git::checkout`
- If provided as a number (parses as `u64`): look up branch by `pr_number` in state
- If omitted: existing TUI selector (unchanged)

### 4. `ez create --from <branch>`

Agents often need to create a branch from a specific base without first navigating to it.

**Behavior:**
- Add `#[arg(long)] from: Option<String>` to `Create`
- If provided: validate `from` is managed or trunk, then proceed as if currently on `from`
- The new branch is created at the tip of `--from`; parent is set to `--from` in state
- Does NOT change the working directory's current branch after creation

**Implementation note:** `create.rs` accepts `from: Option<&str>`; if set, use it as the parent instead of `current_branch()`.

### 5. `ez commit --if-changed`

Agents use `git diff --quiet || ez commit` patterns. This flag makes the conditional native.

**Behavior:**
- `--if-changed` flag on `Commit`
- If set and no staged changes (and `--all` not set): exit 0 silently
- If `--all` is also set: stage all, then check if anything got staged before committing

### 6. `ez pr-edit` opens `$EDITOR` when no flags

Currently a no-op when invoked bare; should open the PR body in an editor.

**Behavior:**
- If no `--title`, `--body`, or `--body-file`: fetch current body via `gh pr view --json body -q .body`
- Write to temp file at `/tmp/ez-pr-<number>.md`
- Launch `$EDITOR` (fallback: `VISUAL`, then `vi`)
- On editor exit: if content changed, call `github::edit_pr`
- If content unchanged: "No changes made"
- If `$EDITOR` not set and `vi` not available: error with actionable message

### 7. `ez draft` and `ez ready`

Toggle PR draft status without going to GitHub.

**Behavior:**
- `ez draft`: marks current branch's PR as draft via `gh pr ready --undo <number>`
- `ez ready`: marks as ready for review via `gh pr ready <number>`
- Both validate: on managed branch, PR exists
- `github.rs`: `set_pr_ready(pr_number: u64, ready: bool) -> Result<()>`

### 8. `ez pr-link`

Print PR URL to **stdout** for piping (`open $(ez pr-link)`).

**Behavior:**
- Load state, get `pr_number` for current branch
- If PR exists in state: construct URL from `repo_name()` + `pr_number`; fall back to `get_pr_status` for the real URL
- Print to stdout with no trailing decoration
- If no PR: exit with error "No PR found — run `ez push` first"

### 9. Structured exit codes

All errors currently exit 1. Agents benefit from being able to branch on failure type.

| Exit code | Meaning |
|-----------|---------|
| 0 | Success |
| 1 | General / unexpected error |
| 2 | GitHub API / `gh` CLI error |
| 3 | Rebase conflict (manual resolution required) |
| 4 | Stale remote ref |
| 5 | Usage error (on trunk, branch not tracked, etc.) |
| 6 | Unstaged changes (use `--autostash` or stash manually) |

**Implementation:**
- `main.rs`: match on `downcast_ref::<EzError>()` before the generic `ui::error` + exit 1
- Only `EzError` variants get specific codes; anyhow chained errors fall through to exit 1

### 10. SKILL.md and CLAUDE.md strengthening

**`SKILL.md`** (repo root): A skill file Claude agents can invoke. Describes the full ez workflow, all commands, and rules for when to use ez vs raw git. Format compatible with superpowers Skill tool.

**Global `~/.claude/CLAUDE.md`**: Add hard prohibitions:
> When `.git/ez/stack.json` exists, NEVER use `git checkout -b`, `git commit` directly, or `gh pr create`. Always use `ez` equivalents.

## Testing

Every new feature requires unit tests:
- `git.rs`: test `stash_push`/`stash_pop` parse logic
- `stack_body.rs`-style pure functions get inline `#[cfg(test)]` unit tests
- `ez status --json` / `ez log --json`: test JSON serialization of known state
- `--if-changed`: test with staged/unstaged combinations
- Exit codes: test that `EzError` variants map to expected codes

---

## Deferred

- CI status in `ez log` — requires `gh run list` per branch, slower, separate feature
- `ez rename` — GitHub PR/branch name coupling makes this complex
- `ez restack --onto` — duplicates `ez move --onto`; document the alias instead
- `ez split`, `ez absorb`, `ez land` — high complexity
